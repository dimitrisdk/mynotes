# Greek Quiz Video Production

## When to use

Use for Greek quiz/trivia videos or bounded proofs where the user expects:

- Greek-only on-screen copy and branding;
- a fixed 2×2 answer-card system;
- a real logo in intro/outro;
- one relevant image or visual motif per question;
- readable typography over image backgrounds;
- countdown pacing and per-second countdown sounds;
- explanation plus a concise fun fact after each reveal;
- mobile readability and a human visual review before delivery.

This skill is for the whole production loop, not only rendering: brief → assets → render → independent QA → human review → release gate.

## Non-negotiable user-facing quality rules

1. **Inspect the actual artifact, not only the renderer or QA script.** Extract and visually inspect the encoded MP4 at intro, question, countdown, reveal, fun-fact, outro, desktop and mobile sizes.
2. **Use the exact approved logo asset.** Never substitute a generated wordmark when the user supplies or approves a logo. Verify the visible pixels show the full logo in intro and outro.
3. **Use one relevant background per question.** Keep the image visible at the edges, but apply a controlled dark scrim and local readability panel. Question-specific imagery must be subordinate to question/cards.
4. **Optical alignment beats coordinate-only alignment.** A script can report equal rectangles and zero baseline delta while the UI still looks wrong. Review 1:1 card crops and mobile frames for perceived centering, glyph shape, badge/text relationship, and long-vs-short answer balance.
5. **Hierarchy must be obvious.** The intended order is: question → answer cards → active countdown. Progress, score, category, difficulty, logo and decoration are secondary. Remove unexplained arrows, duplicate progress bars, oversized score blocks and competing ornaments.
6. **Greek labels must be visibly and audibly correct.** Use actual `Α Β Γ Δ`, not Latin lookalikes or tiny unreadable blobs. In narration, do not synthesize bare glyphs and hope they are pronounced correctly: explicitly write `Άλφα`, `Βήτα`, `Γάμα`, `Δέλτα` in option and reveal scripts. If the voice still mispronounces them, omit spoken letter names rather than ship ambiguous audio. Test single-digit, double-digit and long Greek options.
7. **Countdown audio must be explicit.** If the user requests a cue every second, verify five measurable cue events per question at the intended timestamps. A single final beep is not sufficient. A single final beep is not sufficient.
8. **Reveal must teach.** After the correct card highlight, show a short explanation and one concise, sourced fun fact. Give the fun-fact state enough duration to read and listen.
9. **Do not call a proof approved when voice rights, fact-checking or human review is pending.** Technical PASS and release readiness are separate verdicts.

## Recommended production pipeline

### 1. Freeze the source of truth

Create a versioned manifest containing:

- exact question IDs and count;
- category and difficulty;
- question text, four options, correct index;
- explanation and one fun fact;
- source URL/status for the factual claim;
- scene start/end times;
- background asset path and hash;
- logo asset path and hash;
- audio clip paths, including countdown cues;
- rights status.

Before rendering, assert that question count, scene count, reveal count, audio concat count and visible question IDs agree. Never leave an extra visible question out of the manifest.

### 2. Prepare assets

- Copy approved logo into a versioned project asset directory and hash it.
- Generate or select one question-specific background per question.
- For image backgrounds, keep a dark navy full-frame scrim around 50–65% opacity and a local panel/gradient around 65–78% behind small text. Tune by visual review, not by claiming formal WCAG certification from luminance samples alone.
- Keep backgrounds free of accidental text, watermarks and answer-revealing clues unless deliberately requested. For quiz questions, prefer category-relevant but generic/atmospheric context; the answer cards and reveal are the source of truth. A recognizable answer object (for example a named planet, landmark, instrument, organ, number sequence or chemical symbol) can leak the answer and should be regenerated even when semantically related.
- Generate original/internal countdown cues when required; record their exact durations and timestamps.
- If the user explicitly requests background music, add a restrained, rights-led instrumental bed after the clean voice mix is approved. Loop or extend it to the full encoded duration, keep it subordinate to speech with measurable low-level mixing/ducking, verify its presence in the final mux, and record music rights evidence. Do not silently omit it or add it by default without the user's request.

ComfyUI is appropriate for original background concepts, but generated text is unreliable. Generate visual motifs without text and add exact Greek wordmarks/cards with deterministic text rendering.

### 3. Build the UI hierarchy

During a question scene:

- Put a compact category/difficulty pill above the question.
- Make the question the strongest text block.
- Use a centered fixed 2×2 answer grid.
- Keep countdown visually active only during the answering window.
- Make progress a small `n/6` label or one slim bar, never duplicate both without purpose.
- Keep score small or reveal it after the answer; never let it compete with the question.
- Remove decorative arrows/lines that look like unexplained controls.
- Use a dedicated, calmer fun-fact panel after the reveal; temporarily dim or hide competing timer/score/progress elements.

For the reusable answer card, define geometry once:

- equal card rectangles and gaps;
- equal badge dimensions;
- badge glyph centered using its actual font box;
- answer text fixed to one left anchor per column;
- common vertical visual center;
- adequate padding and gap;
- explicit tests for `7`, `8`, `9`, `10` and long Greek answers.

Then inspect the rendered result optically. Coordinate metrics are evidence, not acceptance by themselves.

### 4. Structure engagement

A bounded proof should normally contain:

```text
intro/logo
→ question + options
→ 5…4…3…2…1 cues
→ correct-answer highlight
→ explanation
→ sourced fun fact
→ score/progress update
→ next question
→ outro/logo + score/comment CTA
```

Keep intro and outro short and Greek-only. Use the official logo clearly, not as an illegible micro-watermark. Avoid casino-like flashes, noisy reward effects and generic stock slides.

### 5. Independent QA gates

Run separate gates and report them separately:

**Content/source QA**
- encoded text matches manifest;
- exact question/option/correct-answer consistency;
- one defensible answer;
- Greek copy and accents are appropriate;
- fun facts have authoritative sources and explicit fact-check status.

**Visual QA**
- inspect actual intro, all question scenes, countdown, reveal, fun fact and outro;
- inspect full-resolution and mobile contact sheets;
- verify exact logo visibility;
- verify per-question background identity;
- inspect optical alignment in 1:1 crops;
- judge hierarchy and clutter as a human, not only via bounding boxes.

**Technical/motion QA**
- ffprobe duration, streams, resolution and frame count;
- full decode with `-xerror`;
- blackdetect and freezedetect as secondary signals;
- decoded-frame motion and transition review using the `video-motion-transition-qa` skill;
- A/V duration delta and audio cue timestamps.

**Rights/release QA**
- logo/image/music/voice provenance;
- commercial voice license evidence;
- generated image and sound status;
- human listening and visual approval;
- no upload until all gates and explicit user approval pass.

A visual QA report must be allowed to say `FAIL` even if technical scripts pass. If the user says the proof looks wrong, treat that as a first-class failure and reset the layout rather than defending coordinate metrics.

## Recovery when a proof is rejected

1. Mark the rejected artifact superseded; do not present it again as acceptable.
2. Stop any remaining render/QA process that may waste time or mutate confusing outputs.
3. Record the concrete visual failures: wrong logo, clutter, poor hierarchy, optical misalignment, unreadable Greek, wrong background, etc.
4. Reset from the last accepted functional source, not from the rejected UI layout.
5. Make one clean, versioned correction pass with a narrowly scoped brief.
6. Perform human visual inspection before reporting PASS.

Do not stack many unverified micro-fixes on top of a failed layout.

## Delivery format

When sending a proof, include:

- direct `MEDIA:/absolute/path/to/file`;
- exact version and purpose (e.g. alignment proof vs background/audio proof);
- SHA-256 and ffprobe facts verified independently;
- what is included and what is deliberately not included;
- blockers such as silent timing bed, draft voice, pending fact-check or pending human approval.

Never imply a proof is approved or publishable when it is only technically rendered.

## User-specific refinements learned from production

- Treat semantic background correctness as a hard gate: if the question asks for the Acropolis/Parthenon, the frame must visibly show recognizable Greek classical architecture; generic historical, Arabic or Middle Eastern buildings are a FAIL. Every question must have a visibly identifiable, question-specific background; do not let an opaque scrim erase it.
- Inspect the supplied logo's alpha/channel state before compositing. If it is opaque/matted and the user wants integration with the video background, create a versioned transparent treatment or exact background-color treatment, then inspect it on the encoded frame. Never accept a visible black/white rectangle around the logo.
- For voice/pacing proofs, use narration for the question and answer/reveal, but normally keep fun facts silent so the viewer can read them. Give the question, answer grid and fun-fact card explicit reading time; do not fill the fun-fact state with narration unless requested.
- Do not add background music by default. First approve voice clarity and pacing with a clean mix; if music is later requested, use a restrained modern instrumental bed with ducking and rights evidence. Avoid synthetic, generic, ethnic or rebetiko beds unless the user explicitly approves the actual listening result.
- Use a focused single-question audio/pacing proof before a long render when the user is still tuning voice and reading time. Label it as a bounded proof, not as evidence that the complete multi-question episode is finished.
- If a full render fails from resource limits, report that honestly and switch to a bounded proof or a lower-memory pipeline; never claim an MP4 exists without verifying the actual file.

## Latest proof-loop lessons (2026-07)

- **Never infer visual correctness from source assets or a QA script.** Inspect the encoded frame: generated backgrounds can contain arrows, pseudo-text, fake logos, culturally wrong architecture, or inset panels. Replace the source rather than hiding a bad artifact with an obvious blur/block.
- **Semantic background correctness is separate from background presence.** A visible image is not sufficient: it must match the question. If the prompt asks for the Acropolis/Parthenon, use recognizable Greek classical architecture; Arabic/Middle Eastern or generic historical buildings are a hard FAIL.
- **Inspect logo alpha before compositing.** An RGB/opaque logo can produce a black/white rectangle. Create a versioned RGBA treatment only after checking that the full eagle/emblem/wordmark/ribbon survived; inspect it against the actual video background.

- **Stray arrows are UI bugs.** Any generated chevron or arrow next to answer cards must be removed unless it has a deliberate, explained interaction meaning. Do not rationalize it as decoration.
- **Audio proof modes must be explicit.** For a reading/pacing proof, put narration on the question and reveal/explanation, keep fun-fact narration off when the user wants reading time, keep countdown cues audible, and leave enough silent duration for the fun-fact text. A silent timing bed is not a speech proof.
- **Approve a bounded proof before scaling.** If the full 1920×1080 render exhausts VPS memory, stop retrying the same job; produce a verified single-question 30–60s proof with the exact voice/timing change, then scale through a lower-memory or desktop pipeline. Never report an MP4 without checking its path, streams, decode and audio presence.
- **Background music is a second mix, not a hidden default.** First approve voice clarity, countdown level and reading time with a clean speech mix. Only then add a restrained modern instrumental bed with ducking and rights evidence; reject generic/synthetic/ethnic/rebetiko beds unless the user approves the actual listening result.

- Treat correct-answer position as a gameplay fairness and retention requirement, not a cosmetic detail. A run dominated by option Α makes the quiz predictable.

## Voice-forward audio mix and loudness gates

- Treat quiet narration as a release-blocking defect, not a subjective postscript. Run a 30–45 second audio proof before a long render whenever voice level or pacing is being changed.
- Use the production target profile: Agapi voice peak **-6 to -12 dBFS**; BGM normally **-18 to -20 dBFS** and ducked to roughly **-22 to -24 dBFS** under speech; SFX/countdown **-10 to -18 dBFS**; final master approximately **-16 to -14 LUFS** with true peak safely below **-1 dBTP**.
- Normalize/boost speech clips before mixing, then meter the encoded final file. Do not infer voice level from raw integer RMS samples or from the renderer's volume constants alone.
- If the final mix peak is just above the chosen ceiling, apply a small final audio-only safety trim and remux with `-c:v copy`; avoid an unnecessary full video rerender. Re-run decode, loudness and peak QA after remux.
- Keep BGM subordinate to the speech with measurable ducking. A low overall LUFS value does not prove that speech is clear; inspect the voice proof and, where possible, meter speech/BGM stems separately.
- Record both proof and final measurements in the episode QA directory. A successful render with quiet voice is not a PASS.

# Quiz Video Production & Visual QA

## Core rule

A render command succeeding is not visual approval. The artifact must pass content, compositing, typography, hierarchy, mobile, timing, rights, and human visual review. Coordinate-only QA is insufficient.

## Workflow

1. Lock the source-of-truth question manifest: IDs, options, correct index, explanations, fun facts, sources, scene timing, and audio clips. Ensure the same question count appears in manifest, renderer, audio concat, contact sheets, and final artifact.
2. Collect/hash exact user-supplied brand assets. If an official logo is supplied, use that exact file in intro/outro; never substitute a generated logo or infer success from generic pixel activity.
3. Generate or select question-specific backgrounds. Keep them semantically relevant but subordinate. Apply a controlled dark scrim and local translucent readability panel so Greek text remains legible; inspect both 1920px and mobile-downscaled frames.
4. Render from clean source assets. Never place new UI over frames from an older rendered proof. Old video frames contain stale text/cards and create duplicate options, blank cards, ghosted labels, and false alignment results.
5. Run a single-question preflight before the bounded proof. Use one-digit, two-digit, and long Greek options. Visually confirm all four options appear exactly once and there are no inherited pixels.
6. Build answer cards as one reusable component: equal card rectangles/gaps, fixed badge size, Greek labels `Α Β Γ Δ`, fixed text left edge, optical vertical centering, consistent line height, and readable contrast. Verify actual 1:1 crops; do not rely only on baseline coordinates.
7. Enforce hierarchy: question → answer cards → active countdown. Category/difficulty, progress, score, logo/watermark, and decoration are subordinate. Hide/dim competing UI during fun facts and reveals.
8. Add engagement states deliberately: intro with logo and clear rules, 5-second countdown with one audible cue per second, answer reveal, concise sourced fun fact, score/progress update, and Greek outro with score/comment CTA.
9. QA the exact artifact: full-resolution frames for intro, question, countdown, reveal, fun fact, and outro; mobile contact sheet; exact answer text count; logo identity; scrim contrast; no stale UI; ffprobe/decode/blackdetect/freezedetect; A/V sync and cue timestamps; rights ledger.

# Cron Reminders

Use this skill when the user wants a reminder, scheduled notification, or other time-based follow-up delivered by Hermes cron.

## Core workflow

### Recurring watchdog jobs
- For repeating checks that should emit the exact message text, prefer `no_agent: true` with a shell script.
- Make the script silent when nothing changed; a quiet tick means success, not failure.
- Persist minimal state in a local file so duplicates are suppressed across runs.
- Use this for feed scans, threshold alerts, and heartbeat-style notifications.
- See `references/watchdog-recurring-jobs.md` for a compact pattern and example.

1. **Normalize the schedule first**
   - Prefer an explicit absolute timestamp for one-shot reminders.
   - Always include the user’s requested timezone when it matters (e.g. `Europe/Athens`).
   - If the request says "tomorrow at HH:MM", compute the actual date from the live current time before creating the job.

2. **Keep the reminder text short and preview-first**
   - Telegram-friendly reminders should be brief, direct, and in the user’s language.
   - For every user-facing cron, make the first line a recognisable mobile preview: `<emoji> **Short topic • status**`.
   - Follow with a date/context subtitle and short bullets; exclude raw JSON, logs, file paths, long URLs, and internal IDs.
   - Avoid extra explanation in the cron prompt itself; put only the required output shape and reminder content.

3. **Create the job and verify immediately**
   - Create the job with `hermes cron create`.
   - Read back `hermes cron list --all` to confirm:
     - the next run time is correct
     - the timezone matches expectations
     - the delivery target is correct

4. **Handle DST when the scheduler runs in UTC**
   - If `cronjob` cannot express `Europe/Athens` (or another required local timezone), do not silently claim a fixed UTC cron is year-round local-time correct.
   - For deterministic/no-agent work, run an hourly wrapper using Python `zoneinfo.ZoneInfo("Europe/Athens")`, emit only at the requested local hour, and persist the sent local date to suppress duplicates.
   - For LLM jobs where hourly execution would waste tokens, either make the seasonal UTC trade-off explicit to the user or adjust the UTC schedule at DST boundaries; never hide the one-hour drift.
   - Test scripts from a directory other than the project root. They should `chdir` to their project directory before opening relative SQLite/database paths.

5. **Replacing a repeat-limited recurring job**
   - When increasing frequency on a job with a finite repeat count (for example `4/52`), do not leave it to expire silently.
   - Create a replacement with the desired recurring schedule and default forever repeat, verify it, then remove the old job by its exact ID.

5. **Watch for off-by-one-day mistakes**
   - If the current local time has already passed the intended reminder time, schedule the next valid day instead of the same date.
   - This is common when a natural-language request is converted to a timestamp without checking the clock.

5. **Edit rather than recreate when correcting schedule drift**
   - Use `hermes cron edit` to fix the schedule, name, delivery target, or reminder prompt.
   - Re-verify after editing.

### Weather comparison in recurring briefings
- When the user asks for a weather comparison, keep the existing briefing job and add a compact `Χθες / Σήμερα / Αύριο` block rather than creating a parallel weather notification.
- Include temperature range, condition/rain and wind for each day when available; add one short practical comparison only when useful.
- Historical “yesterday” observations may be unavailable from a forecast page. Never infer or fabricate them; state that the observed data was unavailable and continue with today/tomorrow.
- Keep the weather block Telegram-mobile friendly and preserve the existing calendar, mail and tasks sections.

### Editing existing daily briefings
- When the user says to stop sending one recurring digest, list jobs first and remove/pause the specific digest job; do not remove unrelated background scans unless asked.
- When the user wants another section added to the already-sent daily briefing, update the existing briefing job instead of creating a parallel morning notification.
- If the new section needs live data, prefer a small deterministic script under `~/.hermes/scripts/` plus a prompt instruction to run it, then verify both the script output and the updated job storage.

## Verification checklist

- The job exists in `hermes cron list --all`.
- The next run matches the user’s requested date/time.
- The reminder content is short enough to be sent cleanly in chat.
- The job is not silently left at the wrong day because the requested time already passed.

## Common pitfalls

- Confusing the user’s timezone with the host timezone.
- Creating a one-shot reminder for a time that has already passed locally.
- Writing a long prompt instead of a short reminder message.
- Forgetting to verify the job after creation or edit.
- Leaving a recurring digest active after the user says they do not want those messages anymore. List jobs, identify the specific digest job, remove or pause it, then verify it disappeared from the list.
- Updating a daily briefing prompt without preserving its existing schedule, delivery, skills, and enabled toolsets. Prefer `cronjob action=update` on the existing job ID rather than creating a parallel daily job.
- For daily briefings that include external task boards, use a small deterministic read-only script to collect the task block, then reference that script in the cron prompt. Verify the script output before updating the job.
- Cron may skip an unpinned LLM job after global model/provider drift to prevent unintended spend. If the job is still desired, pin it explicitly with `cronjob action=update` and a `model` object for the intended provider/model; do not recreate the job and risk losing schedule/delivery settings.
- For `no_agent` script jobs, provider-timeout-looking alerts can actually be script timeouts. Read the saved cron output, run the script endpoint manually with a longer timeout, then patch the script timeout and trigger one `cronjob action=run` to verify.
